# Feedback Loop

## When to Use

- 하네스 파이프라인(Phase 1) 완료 후
- 사용자가 실기기/시뮬레이터에서 앱을 써보고 피드백을 줄 때
- 기존 코드의 UI/UX/버그 수정이 필요할 때

## When NOT to Use

- 신규 기능 개발 (harness-pipeline 사용)
- 대규모 아키텍처 변경 (harness-pipeline 사용)

## How It Works

### 라운드 구성

각 라운드는 하나의 관점에 집중한다:

| 라운드 | 관점 | 집중 포인트 |
|--------|------|------------|
| R1 | 기본 흐름 | CRUD, 기본 네비게이션 |
| R2 | UI / 디자인 | 색상, 폰트, 레이아웃, 다크모드 |
| R3 | UX / 사용성 | 사용 흐름, 햅틱, 토스트, 빈 상태 |
| R4 | 핵심 기능 | 앱 고유 기능 동작 검증 |
| R5 | 위젯/확장 | Widget, Live Activity, Intent 등 |
| R6 | 엣지 케이스 | 빈 데이터, 대량 데이터, 빠른 연타, 백그라운드 |

### 피드백 형식 (사용자 -> AI)

```
[버그] 알람 생성 후 목록에 반영 안 됨
[UI] 알람 행의 시간 텍스트가 잘림
[UX] 삭제 시 확인 모달이 없음
[기능] 조용한 알람 토글이 반응 없음
[개선] 목록에 남은 시간 표시 추가
[스크린샷] (이미지 첨부)
```

### AI 처리 프로세스

**핵심 규칙: 피드백이 여러 개여도 반드시 1개씩 처리. 하나를 끝내고 커밋한 뒤 다음.**

```
[피드백 목록 수신]
       |
  1. 분류 & 우선순위 정렬 (전체를 한 번만)
       |
  +--- 반복 (항목 1개씩) ----+
  |  2. ios-reviewer 에이전트로 수정 위임  |
  |  3. 빌드 확인 (xcodebuild)       |
  |  4. output/ 동기화               |
  |  5. 커밋 (1항목 = 1커밋)          |
  +--- 다음 항목 ---------------+
       |
  6. 응답: 처리 목록 + 확인 안내
```

### 우선순위

크래시 > 기능 미동작 > UI 깨짐 > UX 불편 > 개선 요청

### 커밋 컨벤션

```
feedback R{라운드}-{항목번호}: {한 줄 요약}

- [{분류}] {수정 내용}
```

예: `feedback R2-1: TimePicker 텍스트 색상 수정`

### 피드백 기록

`FEEDBACK_LOG.md`에 기록:

```markdown
## R1: 기본 흐름 (2026-04-23)
| # | 분류 | 피드백 | 처리 | 커밋 |
|---|------|--------|------|------|
| 1 | 버그 | 생성 후 목록 미반영 | loadAlarms 호출 추가 | abc1234 |
```

## Examples

```
/feedback R1 시작 -- 기본 흐름 테스트합니다
/feedback [버그] 알람 생성 후 목록에 안 뜸, [UI] 시간 텍스트 잘림
```
